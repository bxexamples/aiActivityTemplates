* About ByStar, BISOS and Blee

This repo is part of the Libre-Halaal ByStar Digital Ecosystem.

For an overview of ByStar, BISOS and Blee, read:
@/bisos/git/bxRepos/bisos-pip/.github/profile/readme.org

For Concepts and Terminology, import:
@/bisos/panels/bisos-core/modelAndTerminology/_nodeBase_/fullUsagePanel

BISOS (ByStar Internet Services OS) is a unified framework for developing internet
services and software-service continuums, layered on top of Debian.

Blee (BISOS Libre-Halaal Emacs Environment) is the primary usage and development
environment, fully integrated with BISOS.

* COMEEGA -- Collaborative Org-Mode Enhanced Emacs Generalized Authorship

All BISOS and Blee software development uses the COMEEGA syntax. Source files are
a combination of the native programming language and org-mode.

For an overview, import:
@/bisos/git/bxRepos/blee/comeega/README.org

COMEEGA makes heavy use of org-mode dynamic blocks (dblocks) which insert
language-specific and org content into source files. Do not hand-edit content
between =####+BEGIN:= and =####+END:= dblock markers — those are generated.

* poly-dblock

org-mode dynamic blocks are used heavily throughout BISOS and Blee.

For an overview, import:
@/bisos/git/bxRepos/blee/poly-dblock/README.org

All dblock definitions are in =/bisos/blee/env3/dblocks/= — read them when needed.

Dblocks are also used for inline unit testing: a dblock executes a bash command
and captures its output directly in the source file. This is the primary
verification mechanism when no separate test suite exists.

* ByStar Naming Conventions

ByStar uses a combination of CamelCase and Snake conventions, applied consistently
across the digital ecosystem:

- Variable, class, and function names: CamelCase
- Concept collections: tagged with a =prefix_= (snake prefix, CamelCase body)
- File names: follow the pattern of their type (=.cs=, =.pcs=, =.org=, etc.)
- Always use =pathlib= for path manipulations (never =os.path=)
- =bisos-pip= library features are preferable to other public libraries

* CI/CD: RawBisos Vagrant-Based Reconstruction

BISOS and Blee do not use conventional CI/CD pipelines (no GitHub Actions, Jenkins, etc.).

Instead, validation is done through periodic complete VM reconstruction using Vagrant.
This is called RawBisos reconstruction. All bisos-pip packages are exercised in that context.

See: =/bisos/git/bxRepos/bxGenesis/start/README.org= and https://github.com/bxgenesis/start

If the subject of this repo relates to a Capability, it will be validated through
the Capability's Inceptor.

Linting is done at release time using linters in =/bisos/pipx/bin/=
(=pyflakes=, =flake8=, =pylint=, =pycodestyle=, =mypy=).
