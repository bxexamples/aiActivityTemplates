* About BISOS, Blee and ByStar

ByStar (By*) is a Libre-Halaal digital ecosystem — a non-proprietary
autonomy-oriented complete digital ecosystem.

BISOS (ByStar Internet Services Operating System) is the technical
foundation of ByStar. It's a layer on top of Debian that unifies
service-side and usage-side software into a single "software-service
continuum." Its basic unit is a *site* (private git server + registrar),
and its core abstractions are BPOs (portable git-based packages of
data/config/software) and PALS (Possession-Assertable Libre Services).

Blee (BISOS Libre-Halaal Emacs Environment) is the primary usage and
development environment for BISOS — a heavily-integrated Emacs layer.
Blee and BISOS are intertwined.

For concepts and terminology, import:
@/bisos/panels/bisos-core/modelAndTerminology/_nodeBase_/fullUsagePanel

* Programming Conventions

Digital-ecosystem-wide programming conventions apply, not language-specific ones:

- Variable, class, and function names: CamelCase
- Concept collections: tagged with =prefix_= (snake prefix, CamelCase body)

* Python Library Preferences

- Prefer =bisos-pip= library features over other public libraries.
- Always use =pathlib= for path manipulations (never =os.path=).

* Python Typing Hints

Use Python type hints in all PyCS and BISOS modules. For external modules
that have not been using type hints, do not add them.

Linting is done at release time. See the =bisos-pip-release= skill.

* No Multi-Platform Compatibility

Only one of the two latest Debian releases is supported. Windows
compatibility is never required. Backward compatibility is limited to the
two latest releases.

* Git Commits and Git Pushes

Do NOT suggest "commit this" or "push this". The developer performs all
commits and pushes. Instead, when work is complete, say something like:
"You can commit when ready."

* bisos-pip Packages Are Local pip-Installed

All bisos-pip packages (including =bisos.b=) are installed via
=pip install -e= (editable/development mode) from their source repos under
=/bisos/git/auth/bxRepos/bisos-pip/=. The venv at =/bisos/venv/py3/bisos3/=
picks up packages directly from the source tree.

*Do NOT edit files under* =/bisos/venv/py3/bisos3/lib/python3.11/site-packages/=.
Always edit the corresponding source repo:

| Package    | Source repo                                         |
|------------+-----------------------------------------------------|
| bisos.b    | /bisos/git/auth/bxRepos/bisos-pip/b/py3/bisos/b/    |
| bisos.pycs | /bisos/git/auth/bxRepos/bisos-pip/pycs/             |

If the package is installed non-editable and the site-packages copy does
not reflect source changes, reinstall editable:

: pip install -e /bisos/git/auth/bxRepos/bisos-pip/b/py3

* Documentation Hierarchy

During AI-Activity, Claude maintains and updates relevant documentation
at multiple levels:

- =AI-WorkPlan.org= — most immediate; ephemeral, will be org-archived
- =AI-DevStatus.org= — dev-state handoff between sessions/machines
- =README.org= (directory or repo) — human-oriented
- Blee-Panels (=py3/panels/.../fullUsagePanel-en.org=) — most permanent

Update Blee-Panels of bisos-pip and blee packages *first*; point to those
from =/bisos/panels= afterwards.

*Important — =README.rst= is generated, not authored.* Every bisos-pip
package has =README.org= (source, at repo root) and =py3/README.rst= (PyPI
form). =README.rst= is regenerated from =README.org= by
=py3/pypiProc.sh -i fullPrep= (and by =fullPrepBuild forPypi= which calls
it). *Never hand-edit =README.rst=* — edits will be silently overwritten
at release time. If you need to update the README, edit =README.org= and
let =pypiProc.sh= regenerate the RST when releasing. Between releases,
=README.rst= being stale relative to =README.org= is expected and
harmless.
