# Blee-Panel Authorship for bisos-pip Packages

Every bisos-pip package has a **primary Blee-Panel** at:

```
py3/panels/<pkg-name>/_nodeBase_/fullUsagePanel-en.org
```

Blee-Panels are the most permanent and most important form of documentation
in the BISOS/Blee ecosystem â€” more permanent than `README.org`, more
permanent than `AI-DevStatus.org`. They are the developer's primary
in-environment usage surface. When they're wrong, users are wrong.

## The one rule about content

**Do NOT hand-edit content between `####+BEGIN:` and `####+END:` markers.**
The body between markers is *generated by Blee/Emacs dblock machinery* when
the panel is opened in Emacs. Author only:

1. The `####+BEGIN:` header line (signature + `:key value` parameters)
2. The plain-org content *between* dblocks (section text, TODO items, etc.)
3. Placeholder content between markers *only* until Emacs regenerates it

If a dblock body needs to change, update the header parameters. See the
`dblock-authoring` skill for the full mechanics.

## Standard panel skeleton (in order)

1. **File header** â€” one `* B|` line, then standard header dblocks:

   | Dblock                                    | Purpose                                |
   |-------------------------------------------|----------------------------------------|
   | `bxPanel:common/title-plus`               | Panel title + roam metadata            |
   | `blee:bxPanel:topPanelControls`           | Blee navigation controls               |
   | `blee:bxPanel:title`                      | Formatted title line with ByStar logos |
   | `blee:bxPanel:terseTreeNavigator`         | Folding TOC of decedents/siblings/ancestors |

   These are boilerplate â€” copy the exact header from an existing
   bisos-pip panel (e.g. `bisos.startAiActivity`, `bisos.gitist`) and only
   change the `#+title`, `#+roam_key`, and node-path values.

2. **Overview section** â€” a `blee:bxPanel:foldingSection` with
   `:title "Overview" :anchor "Panel"`, followed by one paragraph
   describing what the package does. Also point at the package's
   `README.org` panel entry.

3. **Package Installation section** â€” `blee:bxPanel:foldingSection` titled
   "Pip and Pipx Package Installation", containing two `blee:bxPanel:runResult`
   dblocks:
   - `:command "pipx install bisos.<pkgname>"`
   - `:command "pipx uninstall bisos.<pkgname>"`

   Use `:results "none"` on install/uninstall (no captured stdout â€”
   they're actions).

4. **Main command intro** â€” `blee:bxPanel:foldingSection` for the `.cs`
   executable, containing a `blee:panel:icm:py:intro` dblock. This shows
   the command's self-described usage:
   ```
   :icmName "<pkgname>.cs" :comment "Examples Menu" :afterComment "-- -i examples"
   ```

5. **Feature sections** â€” one `blee:bxPanel:foldingSection` per logical
   group of commands. Each contains either:
   - `blee:bxPanel:runResult` dblocks for **read-only / query commands** â€”
     the command is executed when the dblock is regenerated and the output
     is captured in the file (`:results "stdout"`).
   - `blee:panel:icm:py:cmnd` dblocks for **action commands** â€” a clickable
     button in Blee that runs the command interactively.

   Order: read-only commands before action commands within a section.

6. **User Config section** (if the package uses `bisos.b.userConfig_csu`) â€”
   `blee:bxPanel:foldingSection` titled "User Config --- Persistent Parameters":
   - One `blee:bxPanel:runResult` per persistent param, invoking
     `userConfig_get --parName=<name>` with `:results "stdout"` â€” shows the
     current value inline.
   - One `blee:panel:icm:py:cmnd` per param invoking
     `userConfig_set --parName=<name> --parValue=<value>` for setting it.

7. **Design and Evolution TODOs** â€” `blee:bxPanel:evolution` dblock
   followed by a `blee:bxPanel:foldingSection` titled "Design and Evolution"
   containing org TODO items for planned improvements.

8. **Standard footer dblocks** (also boilerplate â€” copy from an existing
   panel):

   | Dblock                                | Purpose                            |
   |---------------------------------------|------------------------------------|
   | `blee:bxPanel:evolutionMaintainers`   | Bug-reports + maintainer contact   |
   | `blee:bxPanel:footerPanelControls`    | Closing navigation controls        |
   | `blee:bxPanel:footerOrgParams`        | Org-mode local params              |
   | `blee:bxPanel:footerEmacsParams`      | Emacs file-local variables         |

## Dblock choice: `icm:py:cmnd` vs `runResult`

Use this decision table:

| The command is...                     | Dblock                              |
|---------------------------------------|-------------------------------------|
| Read-only, quick, output worth seeing | `blee:bxPanel:runResult :results "stdout"` |
| Action with side effects              | `blee:panel:icm:py:cmnd`            |
| Install / uninstall                   | `blee:bxPanel:runResult :results "none"`   |
| Long-running or interactive           | `blee:panel:icm:py:cmnd`            |

`runResult` bakes captured output into the panel â€” great for
"show me what this produces" documentation. `icm:py:cmnd` is a clickable
button.

## Common dblock header parameters

**`blee:bxPanel:foldingSection`**:
```
:outLevel 0 :sep t :title "<Section Title>" :anchor "<Anchor>" :extraInfo "<subtitle>"
```
Use `:outLevel 0` for top-level sections, `:outLevel 1` for nested. `:sep t`
adds a separator line above; `:sep nil` omits it. `:anchor` becomes an
`<<Anchor>>` target for cross-references.

**`blee:panel:icm:py:cmnd`**:
```
:outLevel 1 :sep t :folding? nil :label "Direct" :icmName "<full command>"
:comment "<one-line description>" :afterComment ""
```
Set `:sep t` on the first cmnd in a section, `:sep nil` on subsequent ones
in the same section. `:folding? nil` keeps the cmnd inline (not collapsed).

**`blee:bxPanel:runResult`**:
```
:outLevel 1 :sep t :command "<full command>" :results "stdout"|"none"
:comment "<description>" :afterComment ""
```
`:results "stdout"` captures the command's stdout into the block body when
the dblock is regenerated. `:results "none"` runs the command but doesn't
capture output.

## What to include vs. exclude

Include only what the package *currently* provides:

- Command-only packages: **no** RO, performer, daemon, or service sections.
- Every `icm:py:cmnd` or `runResult` dblock must map to a real, existing
  invocation. If the command doesn't exist yet, don't add its dblock.
- If the package has `userConfig` params, one row per param â€” no more, no
  less.

## Regenerating panels

Open the panel in Blee and use the "Update Buf Dblocks" panel control
(elisp `org-dblock-update-buffer-bx`). This regenerates every dblock in
the buffer, running each command and capturing output. Save the buffer to
persist.

For pure-Python regeneration of the `b:ai:file/particulars` dblock
specifically, use `py-dblock.cs updateDblocks <file>` (from
`bisos.pyDblock`). But most panel dblocks require Emacs â€” regenerate in
Blee.

## Common pitfalls

- **Panels shipped with stale captured output.** `runResult` bakes stdout
  into the file. If the command's output changes and the panel isn't
  regenerated, the panel lies to users. When behavior changes, regenerate
  the panel and commit the update.
- **Panels shipped without regeneration.** A freshly-authored panel has
  placeholder content inside dblocks until Emacs regenerates them. Always
  open in Blee and regenerate before committing.
- **`_nodeBase_/` symlink** â€” `/bisos/panels/bisos-core/bisos-pip/<pkg>/`
  usually contains a symlink to the package's `py3/panels/<pkg>/_nodeBase_/`.
  Update the package's panel first; the `/bisos/panels/` view follows
  automatically.

## What NOT to do

- Do NOT hand-edit dblock body content â€” regenerate.
- Do NOT include panel sections for commands that don't exist yet.
- Do NOT copy an entire panel from another package without updating the
  header (`#+title`, `#+roam_key`, the "This File" path, the "auto" title
  block). A copy with a wrong `#+roam_key` corrupts the panel graph.
- Do NOT add RO / performer / daemon sections to command-only packages.
